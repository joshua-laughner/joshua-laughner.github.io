Setting up a Python environment for reproducible research

 

What my interest in this is

In my Ph.D. work, I did a lot of work with a decently large algorithm to interpret satellite data. By the time I finished, it was split into six different Git repos. There were:

  • the core code
  • utilities functions closely related to the core code, but used by other projects
  • general utility functions I used in all my projects
  • utilities for working with model data needed as input to the algorithm, and
  • two specialized repos that addressed very specific requirements

So it made a lot of sense to break it up like this - it helped keep the code organized, and meant that if I wanted to share say, just the general utilities with someone else, I could give them just the general utilities repo and not the whole algorithm.

Of course, this did cause another issue, namely it became a bit of a chore to have multiple branches for this project, since I would have to create that same branch in every repo, and switch back and forth between branches in all six repos. I'm a big proponent of the Git flow workflow, so I would (a) need to switch all those branches for every branch to develop a new feature, and (b) would have to make sure to merge all six develop branches into master when it came time to release a new version of the satellite data generated by this algorithm. Sometimes I could get away only having a separate branch in one or two of the repos, but that assumed that I either knew I'd only have to make changes in those repos, or could count on myself to remember that I hadn't branched some of the repos.

While I was in the process of splitting up the code into these six repos, I did try to come up with a better way to do it. The problem was that I didn't just want to tie these six repos together in some way - that would have been easy. I also wanted to be able to have separate versions of these repos for different projects.

  1. I wanted a way
  1. Easy for another scientist to obtain. This means a single repo that can just be cloned or downloaded.
  2. Permits reuse of custom utility repos and allows them to be at different versions for different projects.
  3. Easy to modify those utility repos and have the changes take effect immediately. Basically, we don't want to have to do an extra step beyond just editing the utility to make other modules use that change.
  4. Can push/pull changes to and from those utility repos.
  5. All repos are part of the same history. That is, if you have a few different branches for your project, or if you want to go back to an old commit, you don't want to have to remember to do that in multiple repos.
  6. Compatible with Jupyter notebooks.
