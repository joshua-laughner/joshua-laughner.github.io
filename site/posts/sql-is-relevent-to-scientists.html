When its worth using an SQL database in research

I'd heard about databases for a long time, but for scientific programming, I always thought, "What's the advantage over a nice, simple text file?" The way I saw it:

  • I already knew how to work with text files my my programming languages of choice
  • Text files are easy to view and edit

On the other hand, databases seemed like a pain in the neck:

  • A lot of them need a server program set up to let you access the database.
  • I'd have to learn at least enough SQL to interact with the database (and coming from a Matlab/Python background, SQL takes some getting used to)
  • A database just consists of tables, and I can make a table just fine in a text file as a .csv.

It took literally until the last month of my Ph.D., and getting to talk with a web developer who really understood the benefits of databases, before I ended up taking the plunge, figuring out how to work with them, and learning what makes a database better, in some cases, than a simple text file. Those advantages are:

  1. You don't need a server if you use SQLite3, which is a database management system that just saves your database as a single file. No messing around to set up a server.
  2. It's easy to change a line in the middle of a database. It's hard (or at least annoying) to change one in the middle of a text file.
  3. A database can store different data types - no need to convert to and from strings.
  4. A database intrinsically protects you from some mistakes, and lets you protect yourself from others, for example
    1. Every column has a value, even if that value is null. No worrying about your CSV writer skipping a column.
    2. You can put some simple rules on columns, like the value cannot be null, or every value must be unique, that can help catch some mistakes in data entry.
  5. A database has mechanisms in place to deal with simultaneous edits
